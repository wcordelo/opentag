# OpenTag 2 gap remediation report

Date: 2026-07-14. Scope: the 23 findings in `goal-outputs/opentag-2-gap-audit/gap-audit.md`, evaluated against the current shared worktree after the corrective integration pass. This report describes source and local-test state only. It does not treat an available primitive as a resolution unless the production call site uses it and a regression test exercises the relevant boundary.

## Resolution summary

All 23 original findings have a source-level resolution or, for M1, an explicit and tested implementation choice that satisfies the audit's allowed “chosen equivalent” disposition. The final census is 4 Critical, 8 High, 8 Medium, and 3 Low findings, with no omitted ID. The corrective pass closed the last three report-audit conflicts: staged attachments now resolve through the authenticated harness frontend before container admission; final research delivery now emits Retry, Dig deeper, and Export actions that return through exact durable quick-action admission; and the canonical operations/port documentation now describes those real paths and their activation requirements.

The highest-risk invariants are now enforced at the boundary where they matter. Repository coding and selected-Claude intent fail visibly when the authoritative harness is unavailable instead of reaching AG-UI. A live Slack create reserves a stable client identity, durably confirms the returned timestamp, and performs a bounded exact lookup after an ambiguous create. Final SessionEventDO terminalization occurs inside the exact final-render fence before the Slack request. Recovery losslessly pages output, updates a confirmed live message where possible, retains ownership through ambiguous and rate-limited outcomes, and converts exhausted definitive failures into a retained slow dead letter rather than deleting the last obligation.

Stop, delivery, session, and interaction control now use exact identities throughout. Request-time and alarm-resumed AG-UI Stop require matching accepted and quiescent proof; harness and research controls have their own exact barriers. SessionEventDO multi-row transitions use `transactionSync`. Quick clicks are durably pre-admitted before HTTP 200. Request-time Slack clients share per-channel scheduling and honor `Retry-After`, while the Durable Object alarm owner serializes recovery and persists its next eligible retry. These are source properties backed by local regression tests, not statements about production behavior.

Attachment and context handling now covers the audited path end to end: bounded inline and staged tiers, R2 staging, delayed-file repair, rich history normalization, structured SessionEvent input and tool summaries, prior-thread restaging, authenticated staged-object resolution with size and SHA-256 validation, and exact byte materialization outside the checkout. Requester attribution uses `users.profile.get` with labeled fields and the PR egress policy requires exactly one approved `Prompted by:` line before the write boundary.

No production configuration or remote system was changed during remediation. The remaining work is activation and live verification, listed separately below; those gates are not being represented as completed source work.

## Remaining limitations

The original 23 audit gaps are addressed in source, but several deliberate product/operations limits remain:

- The authoritative harness is source-complete but not enabled by default. With no approved harness activation, repository-coding and selected-Claude turns fail visibly; they do not fall back. This is a safety property, not a usable production coding surface until configured.
- Harness NDJSON is durably recorded, but the Slack coding response is currently posted as one final fenced answer rather than live coding deltas. The `onText` hook remains the future extension point.
- The alternate harness matrix is intentionally narrow: Claude Code is supported; Codex, Amp, and generic reasoning flags are not advertised as active runtimes.
- Staged attachments depend on a shared R2 namespace and access to the original Slack file. The harness frontend enforces a 32 MiB resolved aggregate cap and fails closed on missing objects, size mismatch, digest mismatch, or absent `BLOBS`.
- The request-time Slack rate scheduler is isolate-local by design. Durable Object alarm recovery is a separate sequential owner with persisted deferral timing, so global cross-runtime ordering is achieved by ownership/retry rules rather than one in-memory queue.
- The session viewer is a signed, expiring, read-only JSON surface. It is not a full operator console and does not expose mutation controls.
- Research action cards create ordinary synthetic turns; whether “Export” can produce a particular artifact still depends on the tools and policies available to that subsequent agent turn.
- The harness outbound policy is deliberately GitHub-specific. Other git hosts require their own canonical URL parser, allowlist, credential boundary, branch proof, and API authorization implementation.
- Local fault injection and mock Slack/API tests cannot prove account permissions, Cloudflare service-binding names, R2 bucket identity, Slack app installation, or GitHub token scopes. Those are the reasons the activation checklist remains separate from the source-resolution matrix.

No production traffic or external state is used as evidence in this report. The durable deliverable is the current source, tests, validation script, and this finding-by-finding record.
